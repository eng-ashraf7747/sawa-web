## 1. نبذة عن المشروع

**SAWA** — منصة عروض وخصومات تربط بين مستخدمين (users) وموردين (vendors)، فيها لوحة أدمن كاملة (admin). النموذج: المورد ينشئ عرض → الأدمن يوافق → المستخدم يحجز/يطلب → المورد يسلّم ويسجّل قيمة الفاتورة → سوا تاخد عمولة.

- **الويب:** Next.js — قيد التطوير الفعلي، هو الأساسي حاليًا.
- **الموبايل:** Flutter — جاهز جزئيًا، مش أولوية حاليًا.
- **الحساب المحلي:** `C:\sawa-web` (ويب) | `C:\sawa_app` (موبايل)

## 3. بروتوكول العمل الإلزامي (فريق الويب)

1. ملف واحد فقط في كل رد — الانتظار لتأكيد "تم" قبل الملف التالي.
2. كل خطوة تنتج نتيجة مرئية → اختبار في المتصفح → تأكيد → الانتقال.
3. **Responsive من أول سطر** — لا تعديل لاحق:
   - `px-4 md:px-6 lg:px-8` لكل div رئيسي
   - Bottom Navigation للموبايل + Sidebar للديسكتوب
   - `grid-cols-1 sm:grid-cols-2 lg:grid-cols-3`
4. فصل المنطق عن الواجهة: `hooks/` للمنطق، `lib/` لعمليات Firestore، `components/` للعرض فقط.
5. `npm test` ناجح قبل كل `git push` — بلا استثناء.
6. لا تكرار لنفس الخطأ — كل خطأ يُسجَّل فورًا (قائمة الأخطاء بالأسفل).
7. قبل أي كود جديد يتحقق من: فصل المنطق عن الواجهة / unit test واحد على الأقل / معالجة أخطاء / تحسين المعايير أو الحفاظ عليها.
8. كل قرار معماري مهم يُسجَّل فورًا.
9. لا business logic داخل الشاشات إطلاقًا.
10. `<a href>` تُكتب يدويًا دائمًا (النسخ يحوّلها لرموز خاطئة).

## 4. الأخطاء المكتشفة سابقًا (ممنوع تكرارها)

| # | الخطأ | الحل |
|---|-------|------|
| 1 | `<a href>` يتحول لرمز HTML عند النسخ | يُكتب يدويًا كسطر مستقل قبل href |
| 2 | `uid` غير موجود في `snapshot.data()` | `{ ...snapshot.data(), uid: snapshot.id }` |
| 3 | `imageUrl`/`externalUrl` لا يقبلان `undefined` في Firestore | يُرسل `null` بدلاً من `undefined` |
| 4 | `router.push` لا يعمل خارج سياق Dashboard | استخدم `window.location.href` |
| 5 | Middleware معطل (Firebase Auth يستخدم localStorage) | الحماية Client-side فقط |
| 6 | Sidebar margin لا يعمل مع dynamic classes | `md:mr-64` وليس `md:${cond ? "mr-64":"mr-16"}` |
| 7 | `isAuthorized` يتأخر → query بـ vendorId فارغ | مرّر vendorId فقط بعد `isAuthorized = true` |
| 8 | Firestore يحتاج Composite Index لكل `where + orderBy` | يُنشأ عبر رابط الخطأ في Console |
| 9 | Google Sign-In لا يكتب `role` | يُتحقق ويُعيَّن يدويًا بعد الدخول |
| 10 | `imageUrl?: string` لا تقبل `null` | `imageUrl?: string | null` |

**قرارات تقنية ثابتة:** Cache-Control 7 أيام للصور، Firestore Indexes تُنشأ بلا قلق (مجانية)، `vendorProfiles` منفصلة عن `users`، التواصل المباشر مع المورد مسموح به عمدًا، `LiteralPath` بدل `Path` في PowerShell للمجلدات اللي فيها أقواس.

## 5. البنية المعمارية (Clean Architecture)

```
types/       ← ملف منفصل لكل domain (deal, category, vendor, vendorProfile, transaction, review, analytics...)
lib/         ← عمليات Firestore فقط (auth, categories, deals, users, vendor, storage, analytics, pointsLedger, commissionLedger, exportExcel...)
hooks/       ← منطق منفصل لكل وظيفة (useUser, useDeals, useVendorStats, useExecutiveDashboard, useAsyncAction...)
components/  ← عرض فقط (admin/, vendor/, dashboard/, home/, layout/, auth/)
app/         ← صفحات Next.js فقط
```

**نظام الأدوار:**
- `user`: يرى الفئات والعروض النشطة + نقاط + إحالة + بطاقة المورد
- `vendor`: ينشئ عروض (pending) + ملف تجاري + إحصائيات
- `admin`: يدير كل شيء + يوافق/يرفض + يحوّل مستخدم لمورد

**دورة حياة العرض:** vendor ينشئ → `pending` → admin يوافق → `active` | admin ينشئ مباشرة → `active` | admin يرفض → `rejected` → vendor يعدّل → `pending`

## 10. آخر نقطة توقفنا عندها فعليًا (نهاية السيشن — 30 يونيو، الساعة 20:21)

### السياق:
اكتشفت (بتعمّد) خللًا: بعض الأزرار (تأكيد استلام المورد للفاتورة، تأكيد الاستلام للمستخدم) يمكن الضغط عليها أكثر من مرة قبل أي refresh، مما قد يسبب تنفيذ العملية مرتين. طلبت مراجعة **كل** مواضع الأزرار المشابهة في التطبيق (زر الحجز، الحذف، إلخ) بدون تنفيذ كود فورًا — فقط تحديد المواضع والاتفاق على الحل.

### القرار المتفق عليه (حسب طلبك الصريح، وليس موثّقًا كرد كامل من "فريق الويب" في النص الخام — **نقطة تحتاج تأكيد منك عند الاستكمال**):
1. أي زر إجراء حساس (حجز، حذف مستخدم/مورد/أدمن، إكمال تسليم بقيمة فاتورة) يحتاج:
   - **Guard ضد الضغط المتكرر** أثناء التحميل (loading state).
   - **رسالة تأكيد (confirm dialog)** قبل التنفيذ، خصوصًا لعمليات: الحجز الفوري، الحذف، وتسليم/تأكيد الفاتورة بقيمة خاطئة.

### ما تم تنفيذه فعليًا:
تم إنشاء `hooks/useAsyncAction.ts` — hook عام لمنع تكرار الإجراء أثناء التحميل:

```typescript
// C:\sawa-web\hooks\useAsyncAction.ts
"use client";
import { useState, useCallback } from "react";

interface UseAsyncActionReturn {
  run: (fn: () => Promise<void>) => Promise<void>;
  loading: boolean;
  error: string | null;
  clearError: () => void;
}

export function useAsyncAction(): UseAsyncActionReturn {
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  const run = useCallback(async (fn: () => Promise<void>) => {
    if (loading) return;
    setLoading(true);
    setError(null);
    try {
      await fn();
    } catch (err: any) {
      setError(err?.message ?? "حدث خطأ غير متوقع");
    } finally {
      setLoading(false);
    }
  }, [loading]);

  const clearError = useCallback(() => setError(null), []);
  return { run, loading, error, clearError };
}
```

تمت مراجعته واعتماده من "Code Reviewer" (guard مدمج، useCallback، إدارة أخطاء داخلية). **آخر جملة في السيشن: "أخبرني بالنتيجة" — لسه مستني تأكيدك إنك طبّقته/جربته.**

### الخطوات التالية المباشرة (من حيث توقفنا):
1. تأكيد إن `useAsyncAction.ts` اتعمل وشغال.
2. **تطبيق الـ hook** على الأزرار الحساسة المحدَّدة: زر تأكيد استلام المورد للفاتورة (`VendorBookingCard.tsx`)، وزر تأكيد الاستلام من المستخدم (`BookingCompletionModal.tsx`) — الملفين شُوفوا فعليًا في السيشن.
3. **إضافة confirm dialog** لنفس الأزرار + أزرار الحذف في لوحة الأدمن (مستخدم/مورد/أدمن) + زر الحجز الفوري.
4. مراجعة باقي مواضع الأزرار في التطبيق اللي ممكن يكون فيها نفس مشكلة "الضغط المتكرر" (المهمة دي طُلبت ولسه معلقة — القائمة الكاملة بمواضع الأزرار **مش موجودة بشكل صريح** في نص السيشن، فتحتاج نعيد عملها من الصفر أو تراجعها انت بنفسك في الكود).
5. بعدها: `npm test` + `git commit` + `git push` قبل أي deploy.
